## Context

Session was pure investigation + design discussion, no code changes. Joe wants AUQ (AskUserQuestion) cards to survive being AFK for several hours without the answer getting silently lost. Root cause found: Claude Code's own PreToolUse hook execution has an undocumented default ceiling (roughly 60s-600s per web search of Claude Code hooks docs), NOT this app's own `PROMPT_TIMEOUT_SECS=3600` or curl's `--max-time 3660` (`src-tauri/src/daemon/claude_config.rs` `write_hook_settings()`, ~lines 54-79) - both of those wrongly assumed a 1-hour window was achievable. Claude Code kills the hook process well before that regardless of our own tuning. `src-tauri/src/daemon/hooks_server/permission.rs` currently does an un-timed-out tokio oneshot `rx.await` with no cleanup; `pending_prompts` (`src-tauri/src/daemon/state.rs`) never auto-expire; a late `respond_question` RPC after the hook's owning process died silently fails (send on a dropped channel).

Joe's first instinct was to replace the native AskUserQuestion tool entirely with a hand-rolled parsed-text-question format in the message body (worried native tool relay was the wrong foundation). Investigation found the actual bug is in the *delivery/hold mechanism*, not the question format. Proposed fix (NOT yet validated by /iterate-it, NOT implemented): stop blocking the hook open at all. Have the PreToolUse hook capture the full structured AskUserQuestion tool input (options/multiSelect/headers, already available) and immediately return a quick deny ("answer will follow separately"), ending that turn within seconds - never near any hook ceiling. The question card already stays visible indefinitely in the UI (auto-dismiss-on-timeout was already removed in a prior fix, see `project_auq_never_auto_dismiss` memory). When Joe actually answers - minutes or hours later - the daemon `--resume`s that Claude session with a synthetic user message carrying the formatted Q&A, exactly like any normal follow-up chat turn already works in this app (`src-tauri/src/daemon/lifecycle.rs` handles spawn/`--resume` already).

This design was chosen over the parsed-text-format idea because this app already has proven precedent for message-embedded parsed markers (`<cc-title>`, `<cc-status>`, `<cc-close:starting/done>`) that the resume-synthetic-answer approach reuses architecturally, without losing the native tool's structured data or risking format-drift/ugly literal text on parse failure.
